BASE DE DATOS: qaseduc_calculo_horario
======================================

1. Descripcion general de la base de datos
------------------------------------------
La base de datos `qaseduc_calculo_horario` soporta un sistema de calculo horario y gestion administrativa orientado a colegios y personal contratado. El sistema permite registrar establecimientos, empleados, contratos, distribucion de horarios semanales, opciones de colacion, usuarios de acceso, roles de seguridad y permisos de navegacion por menu.

Desde la aplicacion se observan cuatro areas funcionales principales:

- Gestion organizacional: colegios y su contexto institucional.
- Gestion de personal: empleados, contratos y su carga horaria.
- Planificacion horaria: horarios semanales por empleado y contrato.
- Seguridad y navegacion: usuarios, roles y permisos de menu.

El codigo actual utiliza de forma activa las tablas `colegio`, `empleados`, `contratos_empleado`, `horarios_semanales`, `colacion`, `dias_semana`, `usuarios`, `roles`, `usuario_rol_colegio`, `menu_v` y `usuario_menu_v`. Ademas, el esquema heredado considera `menus` y `usuario_menu`, que representan la version previa del modulo de permisos de interfaz.

5. Explicacion por modulos del sistema
--------------------------------------

5.1. Modulo de colegios y personal
----------------------------------
Tablas:
- `colegio`
- `empleados`
- `contratos_empleado`
- `horarios_semanales`

Descripcion:
Este modulo concentra la operacion principal del sistema. `colegio` define el contexto institucional. `empleados` representa al personal de cada establecimiento. `contratos_empleado` resume la jornada contractual, las horas lectivas, las horas no lectivas y la colacion. `horarios_semanales` descompone esa carga en bloques diarios de trabajo.

Logica funcional:
- Primero se crea o identifica el colegio.
- Luego se registra el empleado asociado al colegio.
- Despues se inserta o actualiza el contrato del empleado.
- Finalmente se registra la distribucion del horario semanal por dias.


5.2. Modulo de apoyo / catalogos
--------------------------------
Tablas:
- `colacion`
- `dias_semana`

Descripcion:
Son tablas auxiliares que entregan valores estandarizados al sistema. `colacion` define opciones de minutos u horarios de colacion. `dias_semana` ordena y presenta los dias que luego se usan para construir la grilla semanal de horarios.

Logica funcional:
- `colacion` alimenta el selector de colacion del resumen.
- `dias_semana` permite listar los dias en orden y combinar dias sin horario cargado mediante `LEFT JOIN`.


5.3. Modulo de seguridad y acceso
---------------------------------
Tablas:
- `usuarios`
- `roles`
- `usuario_rol_colegio`

Descripcion:
Este modulo administra autenticacion, estado de las cuentas y asignacion de perfiles. `usuarios` contiene la cuenta. `roles` define el perfil. `usuario_rol_colegio` enlaza ambos y agrega el contexto de colegio cuando corresponde.

Logica funcional:
- Un usuario puede pertenecer a un colegio principal.
- Un usuario puede tener uno o mas roles segun la tabla de asignacion.
- El sistema usa estas asignaciones para determinar privilegios generales, por ejemplo acceso administrativo.


5.4. Modulo de permisos de interfaz
-----------------------------------
Tablas:
- `menus`
- `menu_v`
- `usuario_menu`
- `usuario_menu_v`

Descripcion:
Este modulo controla la navegacion visible para cada usuario. `menus` y `usuario_menu` corresponden al esquema heredado. `menu_v` y `usuario_menu_v` corresponden a la version activa observada en el codigo actual.

Logica funcional:
- `menu_v` define el catalogo vigente de secciones navegables.
- `usuario_menu_v` almacena si un menu esta permitido o no para un usuario.
- El proyecto actual inicializa permisos base para menus visibles y otorga el menu de usuarios segun rol.
- La presencia de tablas con y sin sufijo `_v` sugiere una migracion progresiva del subsistema de menus.


6. Observaciones tecnicas
-------------------------
- El esquema presenta mezcla de motores MyISAM e InnoDB, lo que dificulta uniformar integridad referencial y operaciones transaccionales.
- Algunas relaciones clave existen solo de forma logica y no como foreign key formal.
- `horarios_semanales` deberia tener constraints formales hacia `empleados` y `contratos_empleado`.
- `empleados`, `usuarios` y `usuario_rol_colegio` deberian tener FK real hacia `colegio`.
- El modulo de menus muestra convivencia entre estructuras heredadas (`menus`, `usuario_menu`) y estructuras vigentes (`menu_v`, `usuario_menu_v`).
- Seria recomendable normalizar mejor ciertas relaciones para evitar duplicidad funcional entre tablas de permisos antiguas y nuevas.
- Seria recomendable migrar completamente las tablas MyISAM a InnoDB para soportar claves foraneas reales, transacciones y borrados consistentes.
- Conviene documentar y formalizar la estrategia de versionado del modulo de menus para evitar ambiguedad entre `menus` y `menu_v`.
- El campo `min_colacion_diaria` se persiste en `contratos_empleado`, mientras `colacion` funciona como catalogo de apoyo; esta decision es valida operativamente, pero deberia documentarse como politica de negocio.
- Las consultas del sistema muestran que muchas relaciones se resuelven por convencion de nombres y por joins manuales desde la aplicacion, lo que incrementa la dependencia del codigo sobre la consistencia de datos.


7. Conclusion final
-------------------
La base de datos `qaseduc_calculo_horario` esta separada en cinco bloques conceptuales bien reconocibles:

- Estructura organizacional: `colegio`
- Personal y contratos: `empleados` y `contratos_empleado`
- Horarios: `horarios_semanales`, con apoyo de `dias_semana` y `colacion`
- Usuarios y roles: `usuarios`, `roles` y `usuario_rol_colegio`
- Permisos de navegacion: `menus`, `menu_v`, `usuario_menu` y `usuario_menu_v`

En terminos funcionales, el nucleo del sistema esta en la cadena colegio -> empleado -> contrato -> horario, mientras que el acceso a la aplicacion se controla con usuarios, roles y permisos de menu. La principal mejora pendiente del esquema es fortalecer la integridad referencial formal para que las relaciones que hoy son logicas queden respaldadas tambien a nivel de base de datos.
